# DBMS Normalization and Functional Dependencies

## Why this topic was chosen

DBMS normalization was chosen because it is one of the most structured and misconception-heavy topics in core database courses.

Students often get stuck on:

- attribute closure
- candidate key derivation
- identifying violating dependencies
- deciding whether a schema satisfies 2NF, 3NF, or BCNF
- checking lossless join
- checking dependency preservation

These mistakes are systematic, which makes them good targets for diagnosis and intervention.

## Why it fits CogniWeave well

This topic is a good fit because:

- it has strong rule-based structure
- it supports a clean concept graph
- mistakes can be tied to very specific sub-skills
- interventions can be step-wise and grounded

Examples of possible interventions:

- closure calculation drill
- identify-the-violating-dependency exercise
- step-by-step normalization exercise
- compare two decompositions and justify the better one
- lossless-join check walkthrough

## What this pack proves

If the system works on DBMS normalization, it shows that the architecture can support formal reasoning tasks, not just practice-heavy or coding-heavy ones.

That makes the project more defensible as a broad adaptive learning engine.

## Why it matters for engineering students

Normalization and functional-dependency reasoning are standard problem areas in DBMS courses and technical preparation.

This pack also strengthens the academic depth of the project while still being very testable.
